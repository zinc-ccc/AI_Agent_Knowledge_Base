# AI 辅助开发与协作方法手册

## 文档定位

这篇文档面向内部分享，沉淀的是可复用的 AI 协作方法，而不是某个单独项目的私有运行细节。

在这个知识库里，文档层次建议这样理解：

- `README.md`
  只负责仓库定位和文档导航。
- `AI_COLLABORATION_AND_METHOD_PLAYBOOK.md`
  负责系统说明“怎么协作、怎么分工、怎么沉淀方法”，可以把它理解为当前知识库里的协作方法正文入口。
- 其他专题文档
  负责承载具体案例、专题学习和单一主题总结。

它重点回答 5 个问题：

- 多个 agent 一起工作时，应该怎么分工
- 什么任务适合谁做
- 如何避免“大家都能做一点，但没人真正负责”
- 怎样把一次次成功经验沉淀成可复用的方法资产
- 完成状态应该如何表达，才不会让协作双方误判

## 一个实用的三角色协作模型

在实际开发和交付里，可以把常见能力拆成 3 类角色：

- `Codex`
  适合做代码库阅读、任务拆解、流程把控、结果复核、规则收口。
- `Claude Code`
  适合做大批量实现、重复修改、草稿实现、搜索并修复这一类执行性很强的工作。
- `Antigravity`
  适合做带真实凭证、真实副作用、真实连接环境的集成操作，例如 Feishu、Bitable、在线联调和运行态验证。

这 3 类角色不是互相替代，而是各自负责最擅长的一段。

## 推荐分工

### Codex 更适合做什么

- 先读真实仓库，再决定怎么做
- 判断任务应该走轻流程还是重流程
- 决定哪些部分适合交给别的 agent
- 复核实现结果和潜在风险
- 把重复经验升级成规则、模板、文档、脚本或 skill

### Claude Code 更适合做什么

- 大范围但边界明确的代码修改
- 重复性较强的重构与替换
- 明确 acceptance criteria 后的实现子任务
- 测试草稿、批量修正、搜索修补类工作

### Antigravity 更适合做什么

- 需要真实账户和运行连接的操作
- Feishu/Bitable 在线读写
- Webhook、WebSocket、消息路由类联调
- 本地 dry-run 无法验证的在线行为检查

## 一条更稳的协作链路

比较稳的协作方式通常是：

1. 先由 `Codex` 读仓库、定范围、定规则。
2. 边界清晰的实现部分交给 `Claude Code` 加速完成。
3. 涉及真实系统联调时，再由 `Antigravity` 做在线执行或验证。
4. 最后回到 `Codex` 收口结果、更新文档、抽取经验。

这样做的好处是：

- 不容易在一开始就把任务做重
- 不容易把在线联调和本地推断混在一起
- 每个 agent 都更像在自己的强项里工作
- 经验更容易被沉淀下来，而不是散在聊天记录里

## 什么时候不要硬上多 agent

并不是所有事情都值得多人协作。
下面这些情况通常直接由一个 agent 处理更高效：

- 只改一两行文案
- 单文件的小修复
- 没有外部依赖的简单整理工作
- 明显是本地即可完成的低风险修改

多 agent 的价值，在于降低复杂任务的失控概率，而不是让简单任务变复杂。

## 最新协作约定：什么叫“真正完成”

在协作里，最容易引发误会的，不是技术实现本身，而是状态表达不清。

为了避免“我以为已经好了，其实只是本地改完了”这种情况，建议默认按下面的口径表达状态：

- `本地已修改`
  说明文件已经改了，但还没有形成正式提交。
- `本地已提交`
  说明改动已经进入本地 Git 历史，但远端未必已更新。
- `内部版已发布`
  说明内容已经推送到内部可见的正式远端分支。
- `公开版已发布`
  说明内容已经推送到对外公开的正式远端分支。

如果用户说的是“提交”“提交吧”“改好了就执行”这一类完成型指令，而没有特别说明 `local only`，更稳妥的默认理解应该是：

1. 本地提交
2. 推送到应该去的正式远端
3. 核对远端提交号
4. 明确汇报当前状态

## 方法沉淀比一次完成更重要

很多团队的真实问题，不是“没人会做”，而是“每次都重新做一次”。

更高效的做法是：

- 发现重复模式
- 识别高频失败点
- 判断哪些内容值得固化
- 选择最轻的落点去沉淀

常见落点包括：

- `AGENTS.md` 规则
- 方法说明文档
- 文档模板
- 操作清单
- 脚本
- skill
- ADR

重点不是多写文档，而是把以后还会再发生的事情，提前变成可复用资产。

## 一个简单的复盘问题集

每完成一个中型以上任务，可以顺手问 4 个问题：

1. 这次哪里重复了？
2. 哪个环节最容易返工？
3. 哪个检查如果更早做，会省掉很多成本？
4. 这次经验最适合沉淀成什么？

候选落点通常只有几种：

- 规则
- 文档
- 模板
- 脚本
- skill

## GitLab 与 GitHub 的关系要讲清楚

如果当前 GitLab 知识库本身已经是脱敏后的分享版本，更合理的理解通常是：

- GitLab 版本
  作为当前可分享内容的主版本，方便内部查看、维护和共建。
- GitHub 版本
  作为更方便外部查看的公开入口镜像。

默认情况下，可以以 GitLab 这份脱敏主版本为基础同步到 GitHub。
只有在个别内容仍然只适合内部传播时，才不进入 GitHub 公开镜像。

## 什么应该留在知识库，什么不应该

适合放进知识库的，通常是：

- 通用协作方法
- 脱敏后的项目经验
- 跨项目可复用的流程说明
- 对更多同事有帮助的实践总结

不适合放进知识库的，通常是：

- 部门私密资料
- 个人部署细节
- 密钥与配置
- 强依赖某个项目路径、机器、服务名的运行文档

## 一句话原则

把 AI 当作协作系统来设计，而不只是把它当作一个会回话的工具。
