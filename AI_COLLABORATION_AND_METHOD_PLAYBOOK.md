# AI 辅助开发与协作方法手册

## 文档定位

这篇文档面向内部分享，沉淀的是可复用的 AI 协作方法，而不是某个单独项目的私有运行细节。

在这个知识库里，文档层次建议这样理解：

- `README.md`
  只负责仓库定位和文档导航。
- `AI_COLLABORATION_AND_METHOD_PLAYBOOK.md`
  负责系统说明“怎么协作、怎么分工、怎么沉淀方法”，可以把它理解为当前知识库里的协作方法正文入口。
- 其他专题文档
  负责承载具体案例、专题学习和单一主题总结。

它重点回答 6 个问题：

- 多个 agent 一起工作时，应该怎么分工
- 什么任务适合谁做
- 为什么当前不少开发者会明显更偏向 `Codex`
- 如何避免“大家都能做一点，但没人真正负责”
- 怎样把一次次成功经验沉淀成可复用的方法资产
- 完成状态应该如何表达，才不会让协作双方误判

## 一个实用的三角色协作模型

在实际开发和交付里，可以把常见能力拆成 3 类角色：

- `Codex`
  更适合做代码库阅读、任务拆解、流程把控、结果复核、规则收口，以及把复杂任务真正推进到“可验证完成”。
- `Claude Code`
  更适合做大批量实现、重复修改、草稿实现、搜索并修复这一类执行性很强的工作。
- `Antigravity`
  更适合做带真实凭证、真实副作用、真实连接环境的集成操作，例如 Feishu、Bitable、在线联调和运行态验证。

这 3 类角色不是互相替代，而是各自负责最擅长的一段。

## 为什么很多人现在会明显更偏向 Codex

如果只从真实使用体验出发，而不是从宣传口径出发，`Codex` 现在确实更容易给人“更好用”的感觉。这种“好用”通常不只是一点，而是几层体验叠加出来的。

### 1. 从代码能力看，输出质量和稳定性更容易让人放心

在真实项目里，很多人最先感受到的差别，还是代码结果本身：

- 更愿意先读仓库，再动手修改
- 更容易围绕真实文件、真实路径、真实约束来工作
- 在中型以上任务里，整体实现更稳，不容易一开始就跑偏
- 对“改完之后怎么验证”这件事更敏感

这会直接体现在结果上：不是只有“能写出来”，而是“更像一个能持续协作的工程结果”。

### 2. 从用户操作体验看，简单、直接、效率高

很多同事会觉得 `Codex` 更顺手，不只是因为模型能力，而是因为交互负担更低：

- 任务入口更直接
- 状态表达更清楚
- 一次对话里更容易把“分析、修改、验证、汇报”串起来
- 同时处理多个线程时，更容易维持节奏

这种体验上的差异，对高频协作非常重要。
很多时候，真正拖慢效率的不是模型答不出来，而是操作链路太绕、状态不透明、重复确认太多。

### 3. 从协作可靠性看，Codex 更像一个真正的执行搭档

一个好用的 agent，不只是“回答像不像”，而是能不能稳定地参与真实工作流。

当前 `Codex` 更突出的地方通常包括：

- 更强调先看真实仓库再判断
- 更容易把任务拆成可以落地执行的步骤
- 更愿意说明“本地改了”“已提交”“已发布”的区别
- 更适合在规则约束下持续协作

这类能力对个人使用很重要，对团队协作更重要，因为它降低了“以为改好了，其实只是说得像改好了”的风险。

### 4. 从并行协作看，Codex 更适合做主控位

如果一个人同时开多个线程、管理多个任务、或者在几个 agent 之间切换，`Codex` 更适合承担主控角色：

- 比较适合统一收口上下文
- 比较适合决定“什么该自己做，什么该交给别的 agent”
- 比较适合在任务结束后顺手沉淀规则和经验

这也是为什么在多 agent 协作里，更推荐把 `Codex` 放在第一位，不是因为别的工具没用，而是因为它更适合做主线控制。

## 这不等于一味否定 Antigravity 和 Claude Code

真实情况通常不是“谁完全替代谁”，而是不同工具在不同环节的性价比不同。

### Antigravity 的价值依然很明确

很多早期项目里的代码、配置和联调痕迹，往往都能看出 `Antigravity` 参与过。这并不奇怪，因为它在一些带真实连接、真实副作用的场景里，本来就很有价值：

- Feishu、Bitable 等在线系统操作
- 需要真实凭证的联调
- 本地静态分析无法替代的运行态验证
- 对接外部系统时的在线排查

如果把它放在对的位置，它依然是很有用的执行器。
问题通常不是它“有没有用”，而是过去有些任务本该先做静态分析和规则收口，却过早进入了在线执行阶段，结果把很多不稳定因素提前放大了。

### Claude Code 仍然适合做高执行性的子任务

`Claude Code` 依然适合做一些边界很清晰、但工作量比较大的实现任务：

- 大范围但边界明确的代码修改
- 重复性较强的重构与替换
- 明确 acceptance criteria 后的实现子任务
- 测试草稿、批量修正、搜索修补类工作

所以更合理的理解，不是“只用一个工具”，而是：

- `Codex` 更适合做主控和收口
- `Claude Code` 更适合做大批量实现加速
- `Antigravity` 更适合做真实在线执行与验证

## 推荐分工：什么软件更适合干什么

如果要给一个更适合团队传播的简化版判断，可以这样理解：

### 更推荐优先打开 Codex 的场景

- 你还没完全看懂仓库，希望先分析再动手
- 任务横跨多个文件或多个模块
- 你担心返工，希望先把边界和风险讲清楚
- 你希望最后能明确知道到底是“改了”“提了”“发了”
- 你想把一次任务顺手沉淀成规则、文档、模板或 skill

### 更推荐交给 Claude Code 的场景

- 你已经把范围定义清楚了
- 你需要批量改很多类似代码
- 你需要一个实现草稿或大规模搜索修补
- 你希望把一个执行性很强的子任务外包出去

### 更推荐交给 Antigravity 的场景

- 你要操作真实 Feishu 或 Bitable
- 你要验证在线行为，而不是本地猜测
- 你要处理 webhook、WebSocket、消息路由这类运行态问题
- 你确认这件事的关键在“真实环境”，而不是“本地代码改造”

## 一条更稳的协作链路

比较稳的协作方式通常是：

1. 先由 `Codex` 读仓库、定范围、定规则。
2. 边界清晰的实现部分交给 `Claude Code` 加速完成。
3. 涉及真实系统联调时，再由 `Antigravity` 做在线执行或验证。
4. 最后回到 `Codex` 收口结果、更新文档、抽取经验。

这样做的好处是：

- 不容易在一开始就把任务做重
- 不容易把在线联调和本地推断混在一起
- 每个 agent 都更像在自己的强项里工作
- 经验更容易被沉淀下来，而不是散在聊天记录里

## 什么人更适合 multi-agent 协作

多 agent 协作不是“越多人越高级”，而是适合那些已经能判断任务边界、知道什么时候该切换工具的人。

更适合上 multi-agent 的，通常是：

- 已经有一定项目经验
- 能区分本地问题、代码问题和在线环境问题
- 愿意做基本的任务拆解
- 知道什么结果需要验证，什么结果只是在猜

如果还处在“先把一件事跑通”阶段，其实先把一个主力工具用顺，比同时开多个 agent 更重要。

## 什么时候不要硬上多 agent

并不是所有事情都值得多人协作。下面这些情况通常直接由一个 agent 处理更高效：

- 只改一两行文案
- 单文件的小修复
- 没有外部依赖的简单整理工作
- 明显是本地即可完成的低风险修改

多 agent 的价值，在于降低复杂任务的失控概率，而不是让简单任务变复杂。

## 最新协作约定：什么叫“真正完成”

在协作里，最容易引发误会的，不是技术实现本身，而是状态表达不清。

为了避免“我以为已经好了，其实只是本地改完了”这种情况，建议默认按下面的口径表达状态：

- `本地已修改`
  说明文件已经改了，但还没有形成正式提交。
- `本地已提交`
  说明改动已经进入本地 Git 历史，但远端未必已更新。
- `内部版已发布`
  说明内容已经推送到内部可见的正式远端分支。
- `公开版已发布`
  说明内容已经推送到对外公开的正式远端分支。

如果用户说的是“提交”“提交吧”“改好了就执行”这一类完成型指令，而没有特别说明 `local only`，更稳妥的默认理解应该是：

1. 本地提交
2. 推送到应该去的正式远端
3. 核对远端提交号
4. 明确汇报当前状态

## 方法沉淀比一次完成更重要

很多团队的真实问题，不是“没人会做”，而是“每次都重新做一次”。

更高效的做法是：

- 发现重复模式
- 识别高频失败点
- 判断哪些内容值得固化
- 选择最轻的落点去沉淀

常见落点包括：

- `AGENTS.md` 规则
- 方法说明文档
- 文档模板
- 操作清单
- 脚本
- skill
- ADR

重点不是多写文档，而是把以后还会再发生的事情，提前变成可复用资产。

## 一个简单的复盘问题集

每完成一个中型以上任务，可以顺手问 4 个问题：

1. 这次哪里重复了？
2. 哪个环节最容易返工？
3. 哪个检查如果更早做，会省掉很多成本？
4. 这次经验最适合沉淀成什么？

候选落点通常只有几种：

- 规则
- 文档
- 模板
- 脚本
- skill

## 什么应该留在知识库，什么不应该

适合放进知识库的，通常是：

- 通用协作方法
- 脱敏后的项目经验
- 跨项目可复用的流程说明
- 对更多同事有帮助的实践总结

不适合放进知识库的，通常是：

- 部门私密资料
- 个人部署细节
- 密钥与配置
- 强依赖某个项目路径、机器、服务名的运行文档

## 一句话原则

把 AI 当作协作系统来设计，而不只是把它当作一个会回话的工具。
