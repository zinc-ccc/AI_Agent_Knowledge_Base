# 以 ArchGuard 为例：如何学习、吸收并沉淀成自己的方法

日期：2026-03-11

## 写这篇文档的目的

这篇文档不是为了教大家“照抄 ArchGuard”。
它更想展示的是：当我们看到一个优秀项目或一套优秀 skill 时，应该怎样学习、怎样吸收、怎样结合自己的项目现实做取舍，最后再沉淀成自己的方法资产。

重点不是 ArchGuard 本身，而是“如何学习一个优秀样本”。

## 触发背景

这次学习的直接触发，来自一位架构师给出的三句话建议：

1. 用 ArchGuard 分析和评价本项目架构。
2. `/proposal-to-plan` 按流程实现以上建议。
3. 执行该计划。

这三句话很有启发，但也不能机械照搬。
对于大型系统、明确架构分层、长期演进的软件项目，这种方式很合适；但对于偏运维、偏集成、偏脚本、偏自动化的项目，如果完全照搬，流程就可能过重。

所以更合理的做法是：

- 先学习它的方法
- 再判断哪些适合自己的项目
- 最后做轻量吸收，而不是生搬硬套

## 我们这次是怎么学习的

### 第一步：先吸收分析，不急着改规则

第一反应不是“把它抄过来”，也不是“立刻更新自己的 skill”，而是先做两件事：

- 用 CLI 真正读取仓库内容，而不是只看 GitHub 页面上的 README
- 先判断它到底哪里强，强在方法、工件、流程，还是某一段提示词

这里有一个很重要的意识转变：

- 不是看到 skill 就去收集 skill
- 而是要追问：它背后的工程方法是什么

### 第二步：不只看 skill 文本，而是看它有没有形成闭环

学习一个方法时，不能只看 `SKILL.md` 写得漂不漂亮，更重要的是看它有没有形成真实闭环，例如：

- skill
- proposal
- plan
- ADR
- code
- tests
- 可查询工件

如果这些东西在仓库里都真实存在，那说明它不是“说得好”，而是“真的在用”。

### 第三步：把“我觉得不错”翻译成“对我项目有什么用”

这一步最关键。很多人学习优秀项目时，容易停在“这个看起来很高级”，但真正有价值的问题应该是：

- 它解决了什么反复出现的问题？
- 它避免了什么失败模式？
- 它有哪些做法可以迁移到我自己的项目？
- 哪些东西只适合它自己，不适合我？

这次吸收里，最值得借鉴的，不是它的技术实现细节，而是下面这些方法：

- skill 不是 magic prompt，而是工程方法的压缩表达
- 好流程必须有工件支撑，而不是只停留在聊天里
- 单一真相入口要靠约束和工件保证，而不是靠口头默契
- 不同工作类型要有不同重量级的 workflow，不要用一个超大 skill 包打天下

### 第四步：反过来审视自己项目还缺什么

学习别人的真正价值，不在“别人有什么”，而在“我因此看清自己缺什么”。

这次复盘后，最直接的收获不是“我们也要做一个 ArchGuard”，而是更清楚地看到：

- 我们需要把方法说明升级成真实工件
- 我们需要把关键约束升级成更明确的规则或 ADR
- 我们需要把单一真相和方法沉淀讲清楚并固化
- 我们需要让 agent 不只会做事，还会在任务后抽取方法资产

## 一个可复用的学习框架

以后遇到别的优秀项目，也可以用同样的 5 步来学：

1. 不先抄做法，先看它解决了什么问题。
2. 不只看提示词，重点看仓库里有没有真实工件闭环。
3. 不只问“它好不好”，要问“它哪些部分能迁移到我这里”。
4. 明确区分“值得学习的原则”和“只适合它自己的实现”。
5. 学完以后，一定要落成自己的规则、文档、模板或 skill，否则学习就停留在聊天层面。

## 对个人思路的启发

这次最大的启发，不是“以后去收集更多 skill”，而是下面这些思维升级：

- 从“我要一个厉害 prompt”，升级到“我要一套可执行的方法”。
- 从“我知道这样做比较好”，升级到“我要让仓库留下证据，证明我们就是这样做的”。
- 从“总结经验”，升级到“提前识别失败模式并做前置约束”。
- 从“解决一次任务”，升级到“把可重复成功的方法产品化”。

## 对 agent 的要求也要升级

如果 agent 只是一个执行器，那它最多帮我们提速。
如果 agent 能在任务后顺手抽取方法、识别可复用规则、提醒补充工件，它才会真正变成团队方法论的一部分。

所以以后对 agent 的期待，应该包括：

- 不只交付结果，也交付方法沉淀建议
- 主动识别哪些经验已经值得固化
- 区分一次性方案和可复用 workflow
- 在中大型任务结束后，做一次简短的方法抽取

## 这次学习过程的示例对话

下面这段不是为了记录闲聊，而是为了让别人看见“一个学习动作是怎么推进的”。

### 对话节选一：先吸收，再判断

> 我之前给你发过一个 skill，现在有了更完整的版本。你吸收一下，看下哪些对我们项目有价值，先吸收分析，再汇报给我，不要直接更新规则。
>
> 因为我也需要学习思路升级；另外，这是 GitHub 的链接，我是希望你用 CLI 去读取，而不是抓取页面的 readme，这没有用。

这一段很重要，因为它体现了一个高质量学习动作：

- 不急着改规则
- 不把 README 当成真实方法来源
- 强制回到仓库本体、真实文件、真实实现

### 对话节选二：从“拿 skill”转向“长出自己的 skill”

> 这个思路确实非常值得学习，这套 skill 也很有价值；但是我不能依赖总别人给我 skill，那我们以后规划和执行项目的时候，要怎么沉淀像这样的 skill？

这句提问特别关键，因为真正的升级，不是学会“拿来用”，而是学会“自己长出来”。

### 对话节选三：从结果导向升级到方法导向

> 你需要办两件事：
> 1. 整理一下本次对话吸收提取的内容，沉淀成一个 md 文档，方便后续分享参考；最理想的是直接展现本次的完整对话，让大家都能看到我的思考过程。
> 2. 你把值得我们学习的内容，加入规则进行更新，然后以后执行的时候，按你本次吸收的内容，来做。

这里体现的是“学习闭环”意识：

- 学完不止口头总结
- 要形成文档
- 要形成规则
- 要在后续执行中真的改变行为

## 一句话结论

学习优秀项目，最重要的不是“复制它”，而是借它看清自己应该如何升级。
